# Contributing Guide

Version: 1.0.0

---

# Purpose

This document defines the development standards for the Garmin FIT
Analysis project.

Although this project is currently maintained by a single developer,
these guidelines ensure that future development remains consistent,
maintainable, and predictable.

Every contribution should improve one or more of the following:

- correctness
- maintainability
- readability
- extensibility
- documentation

---

# Development Philosophy

The project follows several fundamental principles.

Correctness before performance.

Readability before cleverness.

Explicit behavior before implicit behavior.

Documentation before implementation.

Deterministic output over convenience.

Features should solve real problems rather than anticipate unlikely
requirements.

# Coding Standards

Code should be:

small

predictable

well named

testable

modular

Avoid deeply nested logic.

Prefer small functions with one responsibility.

Avoid unnecessary abstraction.

---

# Naming Conventions

Variables

Use descriptive names.

Example

```
average_heart_rate
```

Avoid

```
avg
```

unless context makes the meaning obvious.

Functions

Function names should describe actions.

Examples

```
parse_fit()

normalize_records()

calculate_hr_drift()

write_json()
```

Classes

Class names should describe responsibilities.

Examples

```
FitParser

ActivityBuilder

MetricCalculator

JsonWriter
```

---

# Comments

Code should explain _why_ rather than _what_.

Avoid

```python
i += 1  # Increment i
```

Prefer

```python
# Skip duplicate timestamps generated by paused activities.
```

---

# Formatting

Use the project's configured formatter.

Formatting should never be performed manually when automated tooling is
available.

---

# Type Hints

Public functions should include type hints.

Example

```python
def parse_fit(path: Path) -> Activity:
```

Type information should improve readability.

---

# Error Handling

Errors should be explicit.

Avoid silently ignoring failures.

Recoverable problems should produce warnings.

Fatal problems should produce informative exceptions.

---

# Testing

Every feature should include appropriate tests.

Tests should verify:

correct output

edge cases

invalid input

error handling

schema compliance

Regression tests should be added whenever bugs are fixed.

---

# Determinism

The same input should always produce the same output.

Avoid:

random ordering

time-dependent behavior

locale-dependent formatting

network dependencies

Deterministic behavior is a core project requirement.

---

# Dependencies

Every dependency should satisfy at least one of the following.

Provides essential functionality.

Improves maintainability.

Reduces implementation complexity.

Dependencies should not be introduced solely for convenience.

---

# Performance

Correctness takes precedence over speed.

Optimize only after correctness has been established.

Avoid premature optimization.

---

# Pull Requests

Every contribution should:

have a clear purpose

remain focused

include documentation updates when necessary

avoid unrelated changes

Large changes should be divided into smaller logical commits whenever
possible.

---

# Versioning

The project follows Semantic Versioning.

Major

Breaking changes.

Minor

New functionality.

Patch

Bug fixes and documentation improvements.

---

# Commit Messages

Recommended format

```
type: concise summary
```

Examples

```
feat: add workout parser

fix: normalize Garmin cadence

docs: update schema specification

test: add threshold workout fixtures

refactor: simplify record parser
```

---

# Review Checklist

Before merging changes, verify:

✓ Documentation updated

✓ Tests pass

✓ JSON schema unchanged unless intentional

✓ Output remains deterministic

✓ Code follows project standards

✓ No unnecessary dependencies introduced

---

# Guiding Principle

Every contribution should leave the project more understandable than it
was before.

Code is temporary.

Architecture and documentation should remain useful for years.
